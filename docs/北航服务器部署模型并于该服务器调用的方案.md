# 北航服务器部署模型并于该服务器调用的方案

> **注意：**当前我已经完成了本机 MM 的相关配置，还没有完成的就是第二步当中的**在北航服务器创建环境并部署模型服务**这一步；

## 相关技术栈

- **vLLM**：用于部署模型，并提供 API 接口；
- **LiteLLM**：用于将多个 API 接口包装成一个统一的接口，并提供路由与负载均衡功能；
- **SSH 隧道**：用于将北航服务器的 API 暴露给 MM 服务器。使用 `autossh` 可以实现自动重连，保证稳定性。

## 架构概览

```text
[ MM 服务器 (主控) ]
  ├─ 1. 实验代码 (Client) ----> 请求发给本地 LiteLLM (Port 4000)
  ├─ 2. LiteLLM (Port 4000, 统一路由) 
  │      ├─> 实例 A (本地): localhost:8000 <-- (MM GPU 1)
  │      ├─> 实例 B (远程): localhost:8001 <== SSH隧道 ==> (北航 GPU 4,5 上的 vLLM)
  │      └─> 实例 C (远程): localhost:8002 <== SSH隧道 ==> (北航 GPU 6,7 上的 vLLM)
```

## 一、网络配置

由于北航有两套 vLLM 实例，我们需要在 MM 服务器上一口气建立两条隧道。

**在 MM 服务器执行：**
```bash
# 同时建立两个端口映射：
# 本地 8001 -> 北航 8013 (实例1)
# 本地 8002 -> 北航 8014 (实例2)
autossh -M 0 -f -N \
  -L 8001:localhost:8013 \
  -L 8002:localhost:8014 \
  beihang-tunnel
```

## 三、 MM 服务器：部署本地实例与 LiteLLM

**1. 启动 MM 本地 vLLM (仅使用 GPU 1)：**
见 [vLLM 启用本地模型服务](vLLM%20%E5%90%AF%E7%94%A8%E6%9C%AC%E5%9C%B0%E6%A8%A1%E5%9E%8B%E6%9C%8D%E5%8A%A1.md)

**2. 配置 LiteLLM 统一网关：**
创建 `litellm_config.yaml`。在这里，我们将三个后端全部列出，LiteLLM 会根据负载自动分发。

```yaml
model_list:
  - model_name: Qwen/Qwen2.5-VL-7B-Instruct
    litellm_params:
      model: hosted_vllm/Qwen/Qwen2.5-VL-7B-Instruct
      api_base: "http://localhost:8000/v1"  # 1. 本地 GPU 1
      api_key: "none"
  - model_name: Qwen/Qwen2.5-VL-7B-Instruct
    litellm_params:
      model: hosted_vllm/Qwen/Qwen2.5-VL-7B-Instruct
      api_base: "http://localhost:8001/v1"  # 2. 北航 GPU 4,5
      api_key: "none"
  - model_name: Qwen/Qwen2.5-VL-7B-Instruct
    litellm_params:
      model: hosted_vllm/Qwen/Qwen2.5-VL-7B-Instruct
      api_base: "http://localhost:8002/v1"  # 3. 北航 GPU 6,7
      api_key: "none"

router_settings:
  routing_strategy: least-busy  # 关键：始终将新任务发给当前最闲的实例
  num_retries: 3                # 如果某个实例挂了，自动重试其他实例
```

**3. 启动 LiteLLM：(建议在 tmux 中运行)**
```bash
tmux new -s litellm
conda activate logma-rag-py12
litellm --config litellm_config.yaml --port 4000
```

## 四、 实验调用示例

你的 Python 实验代码现在非常简洁，它认为自己是在调用一个拥有“超强吞吐量”的单一口模型。

```python
from openai import OpenAI

# 只需要连接 MM 服务器的 4000 端口
client = OpenAI(api_key="sk-123", base_url="http://localhost:4000/v1")

def run_experiment(prompt, image_base64):
    response = client.chat.completions.create(
        model="Qwen/Qwen2.5-VL-7B-Instruct", # 配置文件里的 model_name
        messages=[{
            "role": "user",
            "content": [
                {"type": "text", "text": prompt},
                {"type": "image_url", "image_url": {"url": f"data:image/jpeg;base64,{image_base64}"}}
            ]
        }],
        temperature=0.7
    )
    return response.choices[0].message.content
```

## 方案优势结论：

1.  **管理集中化**：所有的路由逻辑、重试机制、超时控制都在 MM 服务器上的一个 `config.yaml` 里完成。
2.  **资源利用最大化**：
    *   MM 的 GPU 0 可以留着做其他小任务或调试。
    *   北航的 4 张卡被切成两个独立实例，能同时并行处理两个请求，互不干扰。
3.  **安全性**：北航服务器依然不需要开启任何额外的公网端口，所有流量都缩在 SSH 隧道内。
4.  **性能**：省去了北航内部的一层 HTTP 转发，延迟更低。

### 运维小贴士：
如果你发现实验跑着跑着变慢了，可以在 MM 服务器上执行 `ps aux | grep ssh`。如果隧道进程断了，重新运行第一步的 `autossh` 命令即可。
