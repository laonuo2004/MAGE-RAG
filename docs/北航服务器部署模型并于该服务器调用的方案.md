# 北航服务器部署模型并于该服务器调用的方案

> **注意：**
> - 当前我已经完成了本机 MM 的相关配置，还没有完成的就是第二步当中的**在北航服务器创建环境并部署模型服务**这一步；
> - 北航服务器只使用后 4 张卡，保留前 4 张卡给其他人；
> - 此外，由于服务器相当于我们借用的，所以不用的时候记得把服务停掉，释放资源；
> - 我们最终保留三档服务：
>   - **throughput 档**：短上下文、高吞吐，用于大多数实验；
>   - **longctx 档**：平衡上下文与吞吐，用于部分长文档问题；
>   - **maxctx 档**：`128000` 全长上下文，用于少量特殊长文档问题；
> - MM 服务器同样只使用 `GPU 1`，`GPU 0` 留给其他人使用。

## 相关技术栈

- **vLLM**：用于部署模型，并提供 API 接口；
- **LiteLLM**：用于将多个 API 接口包装成一个统一的接口，并提供路由与负载均衡功能；
<!-- - **SSH 隧道**：用于将北航服务器的 API 暴露给 MM 服务器。使用 `autossh` 可以实现自动重连，保证稳定性。 -->
**注意我们不再建 SSH 隧道，而是直接访问 IP 地址**

## 架构概览

- MM 本地只部署 **1 个 `128000` 全上下文实例**；
- 北航部署 **2 个 throughput 实例** 与 **2 个 longctx 实例**；
- LiteLLM 把它们都挂在同一个 `model_name` 下，统一转发；
- 为了让长请求能够自动落到长上下文实例，可以把 `num_retries` 调大一些，例如 `10`。

```text
[ MM 服务器 (主控) ]
  ├─ 1. 实验代码 (Client) ----> 请求统一发给本地 LiteLLM (Port 4000)
  ├─ 2. LiteLLM (Port 4000, 统一路由)
  │      ├─> 实例 A (本地): localhost:8000 <-- (MM GPU 1, 128k 专用)
  │      ├─> 实例 B (远程): localhost:8001 <== SSH隧道 ==> (北航 GPU 4 上的 vLLM)
  │      ├─> 实例 C (远程): localhost:8002 <== SSH隧道 ==> (北航 GPU 5 上的 vLLM)
  │      ├─> 实例 D (远程): localhost:8003 <== SSH隧道 ==> (北航 GPU 6 上的 vLLM)
  │      └─> 实例 E (远程): localhost:8004 <== SSH隧道 ==> (北航 GPU 7 上的 vLLM)
```

## 一、网络配置 (已做)

由于北航有 **四套远程 vLLM 实例**，所以我们需要在 MM 服务器上建立四条隧道。

**在 MM 服务器执行：**
```bash
# 本地 8001 -> 北航 8013 (GPU 4)
# 本地 8002 -> 北航 8014 (GPU 5)
# 本地 8003 -> 北航 8015 (GPU 6)
# 本地 8004 -> 北航 8016 (GPU 7)
autossh -M 0 -f -N \
  -L 8001:localhost:8013 \
  -L 8002:localhost:8014 \
  -L 8003:localhost:8015 \
  -L 8004:localhost:8016 \
  beihang-tunnel
```

## 四、 实验调用示例

调用方式很简洁，统一只连一个入口：

```python
from openai import OpenAI

client = OpenAI(
    api_key="sk-123",
    base_url="http://localhost:4000/v1",
)

def run_experiment(prompt, image_base64):
    response = client.chat.completions.create(
        model="Qwen/Qwen2.5-VL-7B-Instruct",
        messages=[{
            "role": "user",
            "content": [
                {"type": "text", "text": prompt},
                {"type": "image_url", "image_url": {"url": f"data:image/jpeg;base64,{image_base64}"}}
            ]
        }],
        temperature=0.7
    )
    return response.choices[0].message.content
```

## 方案优势结论：

1.  **管理集中化**：所有的路由逻辑、重试机制、超时控制都在 MM 服务器上的一个 LiteLLM 配置里完成。
2.  **资源利用最大化**：
    *   MM 的 GPU 1 专门负责 `128000` 长上下文；
    *   北航的 GPU 4/5/6/7 组成扩容池,吞吐更高，也不影响前 4 张卡；
3.  **安全性**：北航服务器依然不需要开启任何额外的公网端口，所有流量都缩在 SSH 隧道内。
4.  **统一调用**：无论短请求还是超长请求，都只需要调用 `localhost:4000` 这一个入口。

### 运维小贴士：

如果你发现实验跑着跑着变慢了，可以按下面顺序排查。

**1. 检查 SSH 隧道是否还在：**
```bash
ss -ltnp | grep -E '8001|8002|8003|8004'
```

理想状态下应该看到 **4 条独立 ssh/autossh 进程**，而不是一条进程同时占 4 个端口。

**2. 检查 MM 侧是否还能访问到北航实例：**
```bash
curl http://localhost:8001/v1/models
curl http://localhost:8002/v1/models
curl http://localhost:8003/v1/models
curl http://localhost:8004/v1/models
```

如果这里全部超时，但你手动执行下面这种临时隧道又能通：

```bash
ssh -fN -L 18001:localhost:8013 beihang-tunnel
curl http://localhost:18001/v1/models
```

那基本可以判定是“常驻隧道进程坏了”，直接把原有 `8001-8004` 隧道全部清掉并按“一端口一进程”重建即可。

**3. 检查 MM 本地实例：**
```bash
curl http://localhost:8000/v1/models
```

**4. 检查 LiteLLM 入口：**
```bash
curl http://localhost:4000/v1/models
```

**5. 检查 tmux 会话：**
```bash
tmux ls
```
